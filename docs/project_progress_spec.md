**ステータス**: ドラフト

## 1. 概要

### 1.1 背景・目的

#### 現状の課題

現在プロジェクトの進捗管理は Excel で行っているが、以下の課題がある。

- **更新が手間**：PL が手動で Excel を更新する必要があり、更新負荷が高い
- **リアルタイム性がない**：PL が更新しない限り他のメンバーは最新の進捗を確認できない
- **進捗把握がPL依存**：メンバー自身が進捗を確認しづらく、各自の当事者意識が生まれにくい
- **横断管理ができない**：上位役職者が複数プロジェクトの進捗を一覧で確認する手段がない

#### 目的

- **GitHub Projects API との連携によるリアルタイム進捗反映**を実現し、PL の更新負荷を最小化する
- **リアルタイムで進捗を共有**することで、PL 以外のメンバーも常に最新の進捗を把握できるようにする
- メンバー各自が進捗を可視化することで、**それぞれの責任感・当事者意識を高める**
- 上位役職者が**複数プロジェクトの進捗を横断して確認**できる環境を整える
- GitHub Projects を利用していないプロジェクト（Backlog 等）には**CSV インポートをサブ機能として提供**する

### 1.4 今後の展望

MVP 以降の段階で以下の拡張を想定している。

| # | 展望 | 説明 |
|---|------|------|
| 1 | **API 自動同期（Webhook / スケジュール同期）** | GitHub Webhook やスケジュール実行で API 同期を自動化し、手動更新ボタン操作なしでも進捗が反映されるようにする |
| 2 | **Backlog API 連携** | Backlog の API にも対応し、Backlog 利用プロジェクトでも CSV インポート不要にする |
| 3 | **全社への展開** | 全プロジェクトが Excel での進捗管理を廃止し、本 Web アプリに一本化する |
| 4 | **見積レビュー会での活用** | 見積レビュー会の場でも本アプリの進捗・ベロシティデータを参照・活用できるようにする |

## 2. 機能要件

### 2.1 GitHub Projects API 連携機能（メイン）

GitHub Projects（v2）の API を直接呼び出し、タスク情報を取得・進捗データに反映するメイン機能。
CSV と異なりファイル操作なしで最新データを取得できる。

#### 2.1.1 認証方式（確定）

- 認証方式は **GitHub OAuth ログイン** を採用する。
- 未ログイン状態ではアプリ機能を利用できない。
- ログイン後、GitHub アカウントの所属組織を検証し、**`@kentem` 所属以外はアクセス拒否**とする。
- 拒否時は「利用対象外アカウント」である旨を明示し、プロジェクト一覧・詳細・同期機能への遷移を禁止する。

#### 2.1.2 取得対象データ

GitHub Projects API（GraphQL）から以下の情報を取得する。

| 項目 | GitHub 上の対応 |
|------|----------------|
| タスクタイトル | Issue / Draft Issue の Title |
| ステータス | Projects の Status フィールド |
| 担当者 | Assignees |
| イテレーション | Projects の Iteration フィールド |
| 予定時間 | Projects のカスタム数値フィールド（PL がマッピング指定）|
| 実施時間 | Projects のカスタム数値フィールド（PL がマッピング指定）|
| ポイント | Projects のカスタム数値フィールド or Labels（`sp:N` 形式）|
| 区分（FE / BE / テスト）| Labels または Projects のカスタムフィールド（PL がマッピング指定）|

#### 2.1.3 フィールドマッピング

プロジェクト登録時に、GitHub 上のフィールドとアプリ内項目の対応を設定する。  
マッピング設定はプロジェクト単位で保存し、以降の同期で再利用する。

| アプリ内項目 | マッピング対象 |
|-------------|---------------|
| タイトル | Issue Title（固定） |
| ステータス | Projects の Status フィールド |
| 完了ステータス値 | API で取得した Status の値の中で「完了」とみなすものを PL が選択（例：`Done`, `Closed`）|
| プルリク中ステータス値 | API で取得した Status の値の中で「プルリク中」とみなすものを PL が選択（例：`In Review`）|
| 担当者 | Assignees（固定） |
| イテレーション | Projects の Iteration フィールド |
| 予定時間 | カスタム数値フィールド名 |
| 実施時間 | カスタム数値フィールド名 |
| ポイント | カスタム数値フィールド名 or Labels（`sp:N` 形式）|
| 区分判定元 | Labels / Projects のカスタムフィールド を PL が選択 |
| 区分値マッピング | 例：`frontend` → `FE`、`backend` → `BE`、`test` → `テスト` |

> 💡 **完了ステータス・プルリク中ステータス**：API で取得した Status 候補値の中から、PL がチェックボックスで選択する方式（CSV のような手入力ではなく候補値ベース）。

> ⚠️ **予定時間とポイントについて**  
> 両者は独立して管理する。進捗率の計算はポイントを使用し、開発効率の計算は予定時間・実施時間を使用する。

#### 2.1.4 同期フロー（MVP）

1. PL がプロジェクト設定で GitHub Projects 接続情報（Project URL または `owner/repo` + Project 番号 + 認証情報）を登録
2. PL がフィールドマッピング・完了/プルリク中ステータス・区分マッピングを設定
3. ダッシュボードまたはプロジェクト詳細画面の **「同期」ボタン** をユーザーが押下
4. アプリが GitHub API を呼び出し、最新のタスク情報を取得
5. 取得データをイテレーションごとに集計し、進捗データを更新
6. 同期時刻を画面に表示（例：「最終同期：2026-04-15 14:32」）

> ⚠️ **MVP では手動更新ボタンのみ**。Webhook やスケジュール同期は将来展望（§1.4）で対応する。

#### 2.1.5 スナップショット保存

API 同期時のタスクデータを **イテレーション単位のスナップショット** として DB に永続化する。

| 保存単位 | 内容 |
|---------|------|
| イテレーション × 同期時刻 | 各タスクの状態（タイトル / ステータス / 担当者 / ポイント / 予定時間 / 実施時間 / 区分）|

これにより以下を実現する。

- 過去イテレーションの進捗履歴を参照可能（読み取り専用）
- 進捗推移グラフ・予想進捗グラフの算出に使用
- API 接続不可時もキャッシュデータで画面表示可能

##### 各指標の計算式

| 指標 | 計算式 | 説明 |
|------|--------|------|
| 進捗率 | 完了ポイント合計 ÷ 目標ポイント合計 | ポイントベースで計算 |
| 開発効率 | 実施時間合計 ÷ 予定時間合計 | 1.0 が予定通り。1.0 超で予定より時間がかかっている |

### 2.2 CSV インポート機能（サブ機能）

> 🟡 **位置づけ**：本機能は **GitHub Projects を利用していないプロジェクト**（Backlog 利用プロジェクト等）向けの **サブ機能** である。
> GitHub Projects 利用プロジェクトでは原則 §2.1 の API 連携を使用する。

#### 2.2.1 対応フォーマット

| ツール | エクスポート方法 | 主要カラム |
|--------|-----------------|------------|
| GitHub Projects | Projects → Export → CSV | `Title`, `Status`, `Labels`, `Assignees`, `Milestone` など |
| Backlog | 課題一覧 → CSV ダウンロード | `課題キー`, `件名`, `状態`, `担当者`, `カテゴリー`, `予定工数` など |

#### 2.2.2 インポートフロー

1. プロジェクト選択 または 新規プロジェクト作成
2. ツール種別を選択（GitHub Projects / Backlog）
3. CSV ファイルをアップロード
4. **カラムマッピング画面**でアプリ内の項目と CSV カラムを対応付け
5. インポート実行 → 進捗データに反映（API 同期と同じくスナップショットとして保存）

#### 2.2.3 カラムマッピング

FE / BE / テスト の分類を含む以下の項目をマッピング設定する。  
マッピング設定はプロジェクト単位で保存し、次回インポート時に再利用できる。

> 💡 **完了ステータスの定義はプロジェクトごとに異なるため、マッピング画面で PL が手入力で設定する。**  
> 例：`Done` / `完了` / `Closed` / `DONE` など複数値をカンマ区切りで登録可能。  
> CSV インポート時に、ステータス列の値がいずれかに一致するタスクを「完了」として扱う。

| アプリ内項目 | 説明 | マッピング例（GitHub） | マッピング例（Backlog）|
|-------------|------|----------------------|----------------------|
| タイトル | タスク名 | `Title` | `件名` |
| ステータス | 完了 / プルリク中 / 未着手の判定元 | `Status` | `状態` |
| 担当者 | メンバー名 | `Assignees` | `担当者` |
| イテレーション | 所属スプリント | `Milestone` | `マイルストーン` |
| 予定時間 | タスクの予定作業時間（h）。1ポイント = 7時間で換算 | `Estimate` | `予定時間` |
| 実施時間 | タスクの実際にかかった時間（h）。開発効率（実施時間 ÷ 予定時間）の算出に使用 | `Time Spent` | `実績時間` |
| ポイント | タスクの計画ポイント。予定時間とは独立して管理 | `Labels`（`sp:N` 形式）| `ストーリーポイント` |
| 区分 | FE / BE / テスト の分類の判定元 | `Labels`（`fe` / `be` / `test` タグ）| `カテゴリー` |

> ⚠️ **区分（FE / BE / テスト）について**  
> インポート時のマッピング画面で「区分」列と CSV カラムを紐付ける。  
> 値の対応（例：`frontend` → `FE`、`backend` → `BE`、`test` → `テスト`）もマッピング画面で設定できる。

> ⚠️ **予定時間とポイントについて**  
> 両者は独立して管理する。進捗率の計算はポイントを使用し、開発効率の計算は予定時間・実施時間を使用する。

### 2.3 プロジェクト管理機能

#### 2.3.1 プロジェクト一覧

- 登録済みプロジェクトをカード形式で一覧表示
- 各カードに「現在のイテレーション進捗」をサマリー表示
- 新規プロジェクト作成 / 編集 / 削除

#### 2.3.2 プロジェクト設定

| 項目 | 説明 |
|------|------|
| プロジェクト名 | 任意の名称 |
| データソース種別 | **GitHub Projects API** / **CSV** から選択 |
| GitHub Projects 接続情報 | データソースが API の場合のみ。Project URL または `owner/repo` + Project 番号 + 認証情報 |
| 参照イテレーションカレンダー | 全社共通の年度カレンダーを選択（§2.4.1）|
| 開始イテレーション | カレンダー上のどのイテレーションから参加するかを選択 |
| プルリク中を進捗に含める | ON / OFF で切り替え（プロジェクト単位）。ON の場合「プルリク中」ステータスのタスクを完了と同じ扱いで進捗率に算入、OFF の場合は未着手と同じ扱いで進捗率に算入しない |
| 総工数（FE / BE / テスト） | データソース同期時に区分ごとのポイント合計を自動集計して表示。手動編集不可（再同期で更新）|

データソース同期時（API 取得 / CSV インポート）に以下を自動集計し、プロジェクト基本情報に表示する。

| 集計単位 | 内容 |
|---------|------|
| プロジェクト全体 | FE / BE / テスト それぞれの総ポイント合計 |
| イテレーションごと | 各イテレーション内の FE / BE / テスト それぞれのポイント合計 |

```
例）プロジェクト基本情報の総工数表示

  ├──────┼──────┼────────┼────────┤
  │ 全体 │ 80pt │  60pt  │  30pt  │
  │ IT#1 │ 20pt │  15pt  │   8pt  │
  │ IT#2 │ 25pt │  20pt  │  10pt  │
  │ IT#3 │ 35pt │  25pt  │  12pt  │
  └──────┴──────┴────────┴────────┘
```

### 2.4 イテレーション管理機能

#### 2.4.1 イテレーションカレンダー（全社共通マスタ）

イテレーションの期間は **開発者・管理者が事前登録する「イテレーションカレンダー」から取得** する。
プロジェクト個別の自動算出は行わない。

- 開発者・管理者が **年度単位で 1 年分のイテレーション期間を一括登録** する
- イテレーション期間は **原則 2 週間** だが、年度開始・年末年始・大型連休・組織イベント等を踏まえ **変則的な期間（短縮 / 延長）も許容** する
- カレンダーは **全プロジェクト共通**。各プロジェクトはカレンダーを参照する（プロジェクトごとの個別期間設定は行わない）
- 各イテレーションは「番号」「開始日」「終了日」を持つ

##### 画面（開発者・管理者専用）

| 項目 | 説明 |
|------|------|
| 年度 | 例：`2026年度` |
| イテレーション一覧 | 番号 / 開始日 / 終了日 / 備考（変則理由など）|
| 操作 | 行追加・編集・削除（変則期間の理由をメモ可）|

#### 2.4.2 プロジェクトとイテレーションの紐付け

- プロジェクト設定で **そのプロジェクトの開始イテレーション**（カレンダー上のどのイテレーションから参加するか）を選択する
- 開始イテレーション以降のすべてのイテレーションが、そのプロジェクトの対象となる
- イテレーション単位で進捗データを表示・切り替え可能
- 過去のイテレーションも参照できる（履歴表示）

### 2.5 チームメンバー管理機能

#### 2.5.1 メンバー情報（イテレーションごとに設定）

| 項目 | 入力方法 | 説明 |
|------|----------|------|
| メンバー名 | 手入力 | 氏名またはアカウント名 |
| 区分 | 選択（FE / BE） | 担当領域 |
| 稼働日数 | 手入力（数値） | そのイテレーション中の稼働日数（詳細は 2.5.2 参照）|
| 目標ポイント | 手入力（数値） | そのイテレーションでの目標ストーリーポイント |
| ベロシティ（実績） | 自動計算 | メンバー個人の消化ポイント ÷ 稼働日数 で自動計算 |
| 目標ポイント | 自動計算 / 手入力 | 稼働日数 × 計画ベロシティ で自動計算。手動上書き可 |

#### 2.5.2 稼働日管理

**アクセス場所：** プロジェクト設定画面内の「稼働日管理」タブ

##### デフォルト稼働日

- 会社全体で 1 つの「イテレーションあたりデフォルト稼働日数」をシステムで固定設定する
- 例：2 週間イテレーションで祝日なしの場合 → `10日`
- このデフォルト値は全プロジェクト・全メンバー共通で適用される

##### 稼働日の個別変更（メンバー × イテレーション単位）

各メンバーの稼働日数はイテレーションごとに個別変更できる。  
有給取得・アサイン率変更・病欠などプロジェクトごとの事情に対応する。

| 項目 | 説明 |
|------|------|
| 稼働日数 | デフォルト値を初期表示。手動で上書き可能 |
| 変更フラグ | デフォルトから変更された場合、**値を赤字で表示** |
| 変更メモ | 変更理由などを自由記述（例：「有給3日取得」「アサイン率50%のため」）|

##### 画面イメージ（稼働日管理テーブル）

```
イテレーション #3  （2026/04/01 〜 2026/04/14）  デフォルト稼働日: 10日

┌──────────┬──────┬──────────────────┬──────────────────────────┐
│ メンバー │ 区分 │ 稼働日数         │ 変更メモ                 │
├──────────┼──────┼──────────────────┼──────────────────────────┤
│ 田中     │ FE   │ 10               │                          │
│ 鈴木     │ FE   │ 🔴 7             │ 有給3日取得              │
│ 佐藤     │ BE   │ 10               │                          │
│ 山田     │ BE   │ 🔴 5             │ 他PJアサイン50%のため    │
└──────────┴──────┴──────────────────┴──────────────────────────┘
```

> - デフォルト値から変更された稼働日数は **赤字** で表示する
> - 変更メモは任意入力。変更した場合に理由を残すことを推奨する
> - 過去イテレーションの稼働日・メモも参照可能（読み取り専用）

### 2.6 進捗表示機能

#### 2.6.1 進捗カード（メイン表示）

各プロジェクトごとに進捗カードを表示する。

```
┌─────────────────────────────────────────────────┐
│  KENTEM TaskHub Core                        [↗]  │
│  Ver 1.8.0  11 イテレーション                    │
│  [████████████████████░░░░░░░░░░░░░]             │
│  予定進捗       実績進捗                         │
│  68.0%          64.5%                           │
└─────────────────────────────────────────────────┘
```

| 表示項目 | 説明 |
|---------|------|
| プロジェクト名 | カードヘッダー左 |
| 詳細画面へのアイコンボタン | カード右上に配置。詳細画面へ遷移 |
| バージョン | プロジェクトのバージョン（例：Ver 1.8.0）|
| 現在のイテレーション | 現在進行中のイテレーション番号（例：11 イテレーション）|
| 進捗バー | 実績ポイント / 目標ポイント の割合（%）|
| 予定進捗 | 経過日数から算出した理想進捗率 |
| 実績進捗 | 完了ポイント / 目標ポイント |

#### 2.6.2 横断ビュー

- 全プロジェクトの進捗カードを一画面に並べて表示

#### 2.6.3 詳細ビュー（プロジェクト個別）

- 全体 / FE / BE の 3 カードを並列表示
- メンバーごとの完了ポイント一覧
- メンバー一覧に以下を表示する：`メンバー名 / 区分 / 稼働日数 / 計画ベロシティ（目標ベロシティ） / ベロシティ（実績） / 開発効率 / 目標ポイント`
- メンバー一覧はイテレーション切替で表示対象を変更できる
- 進捗カードの集計範囲は、イテレーション切替とは別に `選択のみ` / `選択まで累計` を切り替え可能にする
- `選択のみ` は選択中イテレーション単体の値を表示する
- `選択まで累計` は開始イテレーションから選択中イテレーションまでを集計した値を表示する
- 切替対象は「過去イテレーション」と「現在イテレーション」のみ。次イテレーション以降は選択不可（無効表示）
- 稼働日テーブルは「各個人の稼働日数を調整するテーブル」とし、`メンバー名 / 区分 / 稼働日数` を表示する
- イテレーション切替UIには各イテレーションの `開始日 / 終了日` を表示する

##### 閲覧権限（詳細ビュー）

- `user` は **自分が所属しているプロジェクトのみ** 詳細画面を閲覧可能
- `user` が未所属プロジェクトへ直接 URL 遷移した場合は、詳細情報を表示せずアクセス拒否とする
- `admin` / `superAdmin` は全プロジェクト詳細を閲覧可能

##### ユーザー別の過去情報閲覧

- `user` は **自分自身の過去情報**（過去イテレーションの実績・稼働情報）を閲覧可能
- `admin` / `superAdmin` は **各ユーザーの過去情報** を閲覧可能
- 導線は「設定画面のメンバー一覧から遷移」

### 2.8 メンバー詳細画面

#### 2.8.1 遷移

- 設定画面のメンバー一覧から各メンバーの詳細画面へ遷移する

#### 2.8.2 表示内容

| 表示項目 | 説明 |
|---------|------|
| 所属PJ一覧 | 現在および過去に所属していたプロジェクトをすべて表示 |
| 区分 | そのプロジェクト内での担当区分（FE / BE / テスト）|
| 目標ベロシティ | 計画ベロシティ（PL が設定した値）|
| 実績ベロシティ | 消化ポイント ÷ 稼働日数 で算出した値 |
| 開発効率 | 実施時間合計 ÷ 予定時間合計（1.0 が予定通り）|

#### 2.8.3 イテレーション切替

- イテレーションセレクタで表示対象イテレーションを選択できる
- 切替対象は「過去イテレーション」と「現在イテレーション」のみ。次イテレーション以降は選択不可

#### 2.8.4 集計範囲切替

- イテレーション切替とは独立して集計範囲を選択できる

| モード | 説明 |
|--------|------|
| 選択のみ | 選択中イテレーション単体の値を表示 |
| 選択まで累計 | 開始イテレーションから選択中イテレーションまでを集計した値を表示 |

#### 2.8.5 閲覧権限

- `user` は自分自身のメンバー詳細のみ閲覧可能
- `admin` / `superAdmin` は全メンバーのメンバー詳細を閲覧可能

##### 開発効率（イテレーション別）

| 指標 | 定義 | 解釈 |
|------|------|------|
| 開発効率 | 実施時間合計 ÷ 予定時間合計 | 1.0 が予定通り。1.0 超で予定より時間がかかっている |

##### 進捗推移・予想グラフ

折れ線グラフで以下の 3 本の線を表示する。

| 線の種類 | 説明 | 表示区間 | 線種 |
|---------|------|---------|------|
| 実績進捗率 | 各イテレーション終了時点の実際の進捗率 | 開始〜現在イテレーション | **実線** |
| 理想進捗率 | イテレーション数に応じた均等な理想ライン | **全イテレーション（開始〜最終）** | **実線** |
| 予想進捗率 | 全イテレーション（過去〜現在）の平均ベロシティをもとに算出した予測ライン | 現在イテレーション〜将来 | **点線** |

**グラフ表示ルール**

| ルール | 詳細 |
|--------|------|
| X 軸 | プロジェクトの全イテレーション（開始〜最終）を表示する |
| 理想（予定）線 | 全イテレーション分を引く |
| 実績線 | 現在イテレーションまで引く（未来のイテレーションは表示しない）|
| 予想線 | 現在イテレーションの **実績値** を起点として引く（実績線と接続）。以降の未来イテレーションは `predictedRate` を使用する |

**予想進捗の計算方法**

```
平均ベロシティ = 全イテレーション（過去〜現在）の完了ポイント合計 ÷ イテレーション数
予想完了ポイント（次イテレーション以降）= 現在の完了ポイント + 平均ベロシティ × 残イテレーション数
予想進捗率 = 予想完了ポイント ÷ 目標ポイント合計
```

予想進捗率の算出は、常に「全イテレーション（過去〜現在）」を母数として行う。

## 3. 運用フロー

### 3.1 基本的な運用の流れ

```
① PL がプロジェクト初期設定を登録（データソース種別を選択）
        ↓
② PL が全イテレーションの計画をまとめて登録（初回のみ）
        ↓
③ データソースから同期　← イテレーションごとに繰り返す
   ├─ API（メイン）：「同期」ボタンを押下
   └─ CSV（サブ）：CSV をアップロード（GitHub未利用 / Backlog 利用プロジェクト）
        ↓
④ メンバーが進捗ダッシュボードを確認
        ↓
⑤ イテレーション終了 → ③ へ戻る
```

### 3.2 PL の操作（プロジェクト初期設定）

プロジェクト開始時に PL が以下を登録する。

#### ① プロジェクト基本情報

| 項目 | 説明 |
|------|------|
| プロジェクト名 | 任意の名称 |
| データソース種別 | GitHub Projects API / CSV |
| GitHub Projects 接続情報 | データソースが API の場合のみ。Project URL または `owner/repo` + Project 番号 + 認証情報 |
| 参照イテレーションカレンダー | 全社共通の年度カレンダー（管理者が事前登録）から選択 |
| 開始イテレーション | カレンダー上のどのイテレーションから参加するかを選択 |

#### ② チームメンバー登録

| 項目 | 入力者 | 説明 |
|------|--------|------|
| メンバー名 | PL | データソース上の担当者名と一致させる |
| 区分 | PL | FE / BE を選択 |
| 計画ベロシティ | PL | メンバーごとの処理能力の指標（単位なし）。稼働日数と掛け合わせて目標ポイントを算出する |

#### ③ イテレーション計画登録（初回のみ・全イテレーションまとめて登録）

| 項目 | 入力者 | 説明 |
|------|--------|------|
| 稼働日数 | PL | そのイテレーション中の各メンバーの稼働日数。初期値は会社で定められたデフォルト稼働日数が自動入力される |
| 目標ポイント | PL | 稼働日数 × 計画ベロシティ で自動計算。手動上書きも可 |

#### ④ データソース別マッピング設定（初回のみ）

##### A) GitHub Projects API の場合（メイン）

| 項目 | 入力者 | 説明 |
|------|--------|------|
| フィールドマッピング | PL | アプリ項目（予定時間 / 実施時間 / ポイント など）と Projects のカスタムフィールドを対応付け |
| 完了ステータス値 | PL | API で取得した Status 候補値の中から「完了」とみなすものを選択（例：`Done`, `Closed`）|
| プルリク中ステータス値 | PL | API で取得した Status 候補値の中から「プルリク中」とみなすものを選択（例：`In Review`）|
| 区分判定元 | PL | Labels / Projects のカスタムフィールド のいずれかを選択 |
| 区分値マッピング | PL | 区分判定元の値と FE / BE / テストの対応を設定 |

##### B) CSV インポートの場合（サブ：GitHub未利用 / Backlog）

| 項目 | 入力者 | 説明 |
|------|--------|------|
| カラムマッピング | PL | CSV 列とアプリ項目を対応付け |
| 完了ステータス定義 | PL | 「完了」とみなすステータス値を入力（例：`Done, 完了, Closed`）|
| プルリク中ステータス定義 | PL | 「プルリク中」とみなすステータス値を入力（例：`In Review, レビュー中`）|
| FE / BE 値マッピング | PL | CSV 上の値と FE / BE の対応を設定 |

> ✅ マッピング設定はプロジェクト単位で保存されるため、2 回目以降の同期は「同期」ボタン押下（API）または CSV 選択（CSV）だけでよい。

## 7. データモデル（概要）

```
SystemConfig                             ← 会社全体の固定設定
  └── default_working_days_per_iteration ← デフォルト稼働日数（システム管理者が設定）

IterationCalendar                        ← 全社共通・年度ごとのイテレーションマスタ（管理者が登録）
  ├── id
  ├── fiscal_year                        ← 年度（例：2026）
  ├── name                               ← 任意名（例：「2026年度カレンダー」）
  └── Entries[]                          ← 各イテレーションの期間（変則期間も可）
        ├── id
        ├── number                       ← イテレーション番号（1, 2, 3, ...）
        ├── start_date / end_date        ← 開始日・終了日（管理者が登録、変則対応）
        └── memo                         ← 変則理由などの自由記述

Project
  ├── id
  ├── name
  ├── data_source_type                   ← api_github / csv
  ├── iteration_calendar_id              ← 参照する全社カレンダー
  ├── start_entry_id                     ← カレンダー上の開始イテレーション
  ├── ApiConnection                      ← data_source_type = api_github の場合のみ
  │     ├── project_url                  ← GitHub Projects URL
  │     ├── owner / repo
  │     ├── project_number
  │     └── auth_credential_ref          ← 認証情報への参照（保管方法は §8 TBD）
  ├── FieldMapping                       ← マッピング設定（プロジェクト単位）。API/CSV 両対応
  │     ├── source_kind (api / csv)
  │     ├── title_source                 ← API: 固定 / CSV: カラム名
  │     ├── status_source
  │     ├── assignee_source
  │     ├── iteration_source
  │     ├── estimated_hours_source
  │     ├── actual_hours_source
  │     ├── points_source                ← API: フィールド名 or Labels(`sp:N`) / CSV: カラム名
  │     ├── division_source_kind         ← labels / custom_field / csv_column
  │     ├── division_source_name         ← フィールド名 or カラム名（Labels の場合は無視）
  │     ├── done_statuses[]              ← 「完了」とみなす Status 値リスト
  │     ├── in_review_statuses[]         ← 「プルリク中」とみなす Status 値リスト
  │     ├── include_in_review_in_progress ← プルリク中を進捗に含めるか
  │     ├── fe_values[]                  ← FE とみなす値リスト
  │     ├── be_values[]                  ← BE とみなす値リスト
  │     └── test_values[]                ← テストとみなす値リスト
  ├── Members[]                          ← プロジェクト全体のメンバーマスタ
  │     ├── name
  │     ├── division (FE / BE / テスト)
  │     └── planned_velocity             ← 計画ベロシティ（PL が設定）
  ├── TotalPoints                        ← 同期時に自動集計・更新
  │     ├── fe_total
  │     ├── be_total
  │     └── test_total
  └── ProjectIterationStates[]           ← カレンダーエントリごとのプロジェクト状態
        ├── iteration_calendar_entry_id  ← 参照するカレンダーエントリ
        ├── fe_points_total              ← そのイテレーションの FE ポイント合計
        ├── be_points_total              ← そのイテレーションの BE ポイント合計
        ├── test_points_total            ← そのイテレーションのテストポイント合計
        ├── IterationMembers[]           ← イテレーションごとの稼働情報
        │     ├── member_id
        │     ├── working_days
        │     ├── is_working_days_modified
        │     ├── working_days_memo
        │     └── target_points
        └── SyncSnapshots[]              ← API 同期 / CSV インポートの実行ごとに保存
              ├── synced_at              ← 同期実行時刻
              ├── source_kind (api / csv)
              └── Tasks[]                ← この同期時点でのタスク一覧
                    ├── external_id      ← Issue 番号 / 課題キーなど
                    ├── title
                    ├── status
                    ├── points           ← 計画ポイント
                    ├── estimated_hours
                    ├── actual_hours
                    ├── assignee
                    └── division (FE / BE / テスト / unclassified)
```

> 💡 **イテレーションエンティティの統合**：従来 `Project.Iterations[]` として持っていた「番号・開始日・終了日」は `IterationCalendar.Entries[]` に統合し、Project 側はカレンダーエントリへの参照（`ProjectIterationStates[]`）で集計値・稼働情報・スナップショットを保持する。

> 💡 **タスクはスナップショット単位で保存**：同じイテレーション内で複数回同期した場合、最新スナップショットを進捗計算に使用しつつ、過去スナップショットも履歴として参照できる。

## 10. OAuth スコープ不足時の再同意フロー（追記）

GitHub ログインを導入していても、Projects API の利用に必要な権限（`read:project` または `project`）が未付与のケースがある。
そのため、以下の運用を必須とする。

### 10.1 基本方針

- ログイン成功 = Projects API 利用可能、とはみなさない
- 初回API実行時にスコープ不足を検知する
- スコープ不足時はユーザーに再同意を促し、必要スコープで再認可させる

### 10.2 必須スコープ

- 参照のみ: `read:project`
- 更新を含む: `project`

### 10.3 実装要件

1. ログイン後、Projectメタ情報取得APIを実行して疎通確認する
2. GraphQLエラーに `INSUFFICIENT_SCOPES` が含まれる場合、専用エラーUIを表示する
3. エラーUIでは不足スコープ名を表示し、再認可導線（再ログイン/再同意）を提供する
4. 再認可完了後に同じAPIを再試行する

### 10.4 UIメッセージ例

- 「GitHub Projects の参照権限が不足しています。`read:project` 権限で再度連携してください。」

### 10.5 ログ要件

- スコープ不足発生時は以下を記録する
  - login（user/organization）
  - project number
  - 発生日時
  - GraphQL error type（`INSUFFICIENT_SCOPES`）
  - 不足スコープ

### 10.6 トークン管理方針（セキュリティ要件）

`API確認` 機能で GitHub API を呼ぶ際は、固定PATのサーバー保存を前提にしない。
以下を必須要件とする。

1. ユーザーのOAuthログイン後トークンを、API確認時のみサーバー側で一時利用する
2. トークンはブラウザへ返却しない（APIレスポンスに含めない）
3. セッションは `httpOnly` Cookie 等で管理し、フロントJSから直接参照できない形にする
4. DBへの長期保存を避け、短寿命トークンを優先する
5. スコープは最小権限（参照のみなら `read:project`）に限定する
6. CSRF対策を実装する
7. アプリログにトークン文字列を出力しない（マスクまたは非出力）

> 補足: セッション利用自体は一般的な安全手段であり、危険性は保存方式と露出方法に依存する。
