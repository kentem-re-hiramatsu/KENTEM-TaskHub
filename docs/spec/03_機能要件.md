## 2. 機能要件

### 2.1 GitHub Projects API 連携機能（メイン）

GitHub Projects（v2）の API を直接呼び出し、タスク情報を取得・進捗データに反映するメイン機能。
CSV と異なりファイル操作なしで最新データを取得できる。

#### 2.1.1 認証方式（確定）

- 認証方式は **GitHub OAuth ログイン** を採用する。
- 未ログイン状態ではアプリ機能を利用できない。
- ログイン後、GitHub アカウントの所属組織を検証し、**`@kentem` 所属以外はアクセス拒否**とする。
- 拒否時は「利用対象外アカウント」である旨を明示し、プロジェクト一覧・詳細・同期機能への遷移を禁止する。

#### 2.1.2 取得対象データ

GitHub Projects API（GraphQL）から以下の情報を取得する。

| 項目 | GitHub 上の対応 |
|------|----------------|
| タスクタイトル | Issue / Draft Issue の Title |
| ステータス | Projects の Status フィールド |
| 担当者 | Assignees |
| イテレーション | Projects の Iteration フィールド |
| 予定時間 | Projects のカスタム数値フィールド（PL がマッピング指定）|
| 実施時間 | Projects のカスタム数値フィールド（PL がマッピング指定）|
| ポイント | Projects のカスタム数値フィールド or Labels（`sp:N` 形式）|
| 区分（FE / BE / テスト）| Labels または Projects のカスタムフィールド（PL がマッピング指定）|

#### 2.1.3 フィールドマッピング

プロジェクト登録時に、GitHub 上のフィールドとアプリ内項目の対応を設定する。  
マッピング設定はプロジェクト単位で保存し、以降の同期で再利用する。

| アプリ内項目 | マッピング対象 |
|-------------|---------------|
| タイトル | Issue Title（固定） |
| ステータス | Projects の Status フィールド |
| 完了ステータス値 | API で取得した Status の値の中で「完了」とみなすものを PL が選択（例：`Done`, `Closed`）|
| プルリク中ステータス値 | API で取得した Status の値の中で「プルリク中」とみなすものを PL が選択（例：`In Review`）|
| 担当者 | Assignees（固定） |
| イテレーション | Projects の Iteration フィールド |
| 予定時間 | カスタム数値フィールド名 |
| 実施時間 | カスタム数値フィールド名 |
| ポイント | カスタム数値フィールド名 or Labels（`sp:N` 形式）|
| 区分判定元 | Labels / Projects のカスタムフィールド を PL が選択 |
| 区分値マッピング | 例：`frontend` → `FE`、`backend` → `BE`、`test` → `テスト` |

> 💡 **完了ステータス・プルリク中ステータス**：API で取得した Status 候補値の中から、PL がチェックボックスで選択する方式（CSV のような手入力ではなく候補値ベース）。

> ⚠️ **予定時間とポイントについて**  
> 両者は独立して管理する。進捗率の計算はポイントを使用し、開発効率の計算は予定時間・実施時間を使用する。

#### 2.1.4 同期フロー（MVP）

1. PL がプロジェクト設定で GitHub Projects 接続情報（Project URL または `owner/repo` + Project 番号 + 認証情報）を登録
2. PL がフィールドマッピング・完了/プルリク中ステータス・区分マッピングを設定
3. ダッシュボードまたはプロジェクト詳細画面の **「同期」ボタン** をユーザーが押下
4. アプリが GitHub API を呼び出し、最新のタスク情報を取得
5. 取得データをイテレーションごとに集計し、進捗データを更新
6. 同期時刻を画面に表示（例：「最終同期：2026-04-15 14:32」）

> ⚠️ **MVP では手動更新ボタンのみ**。Webhook やスケジュール同期は将来展望（§1.4）で対応する。

#### 2.1.5 スナップショット保存

API 同期時のタスクデータを **イテレーション単位のスナップショット** として DB に永続化する。

| 保存単位 | 内容 |
|---------|------|
| イテレーション × 同期時刻 | 各タスクの状態（タイトル / ステータス / 担当者 / ポイント / 予定時間 / 実施時間 / 区分）|

これにより以下を実現する。

- 過去イテレーションの進捗履歴を参照可能（読み取り専用）
- 進捗推移グラフ・予想進捗グラフの算出に使用
- API 接続不可時もキャッシュデータで画面表示可能

##### 各指標の計算式

| 指標 | 計算式 | 説明 |
|------|--------|------|
| 進捗率 | 完了ポイント合計 ÷ 目標ポイント合計 | ポイントベースで計算 |
| 開発効率 | 実施時間合計 ÷ 予定時間合計 | 1.0 が予定通り。1.0 超で予定より時間がかかっている |

### 2.2 CSV インポート機能（サブ機能）

> 🟡 **位置づけ**：本機能は **GitHub Projects を利用していないプロジェクト**（Backlog 利用プロジェクト等）向けの **サブ機能** である。
> GitHub Projects 利用プロジェクトでは原則 §2.1 の API 連携を使用する。

#### 2.2.1 対応フォーマット

| ツール | エクスポート方法 | 主要カラム |
|--------|-----------------|------------|
| GitHub Projects | Projects → Export → CSV | `Title`, `Status`, `Labels`, `Assignees`, `Milestone` など |
| Backlog | 課題一覧 → CSV ダウンロード | `課題キー`, `件名`, `状態`, `担当者`, `カテゴリー`, `予定工数` など |

#### 2.2.2 インポートフロー

1. プロジェクト選択 または 新規プロジェクト作成
2. ツール種別を選択（GitHub Projects / Backlog）
3. CSV ファイルをアップロード
4. **カラムマッピング画面**でアプリ内の項目と CSV カラムを対応付け
5. インポート実行 → 進捗データに反映（API 同期と同じくスナップショットとして保存）

#### 2.2.3 カラムマッピング

FE / BE / テスト の分類を含む以下の項目をマッピング設定する。  
マッピング設定はプロジェクト単位で保存し、次回インポート時に再利用できる。

> 💡 **完了ステータスの定義はプロジェクトごとに異なるため、マッピング画面で PL が手入力で設定する。**  
> 例：`Done` / `完了` / `Closed` / `DONE` など複数値をカンマ区切りで登録可能。  
> CSV インポート時に、ステータス列の値がいずれかに一致するタスクを「完了」として扱う。

| アプリ内項目 | 説明 | マッピング例（GitHub） | マッピング例（Backlog）|
|-------------|------|----------------------|----------------------|
| タイトル | タスク名 | `Title` | `件名` |
| ステータス | 完了 / プルリク中 / 未着手の判定元 | `Status` | `状態` |
| 担当者 | メンバー名 | `Assignees` | `担当者` |
| イテレーション | 所属スプリント | `Milestone` | `マイルストーン` |
| 予定時間 | タスクの予定作業時間（h）。1ポイント = 7時間で換算 | `Estimate` | `予定時間` |
| 実施時間 | タスクの実際にかかった時間（h）。開発効率（実施時間 ÷ 予定時間）の算出に使用 | `Time Spent` | `実績時間` |
| ポイント | タスクの計画ポイント。予定時間とは独立して管理 | `Labels`（`sp:N` 形式）| `ストーリーポイント` |
| 区分 | FE / BE / テスト の分類の判定元 | `Labels`（`fe` / `be` / `test` タグ）| `カテゴリー` |

> ⚠️ **区分（FE / BE / テスト）について**  
> インポート時のマッピング画面で「区分」列と CSV カラムを紐付ける。  
> 値の対応（例：`frontend` → `FE`、`backend` → `BE`、`test` → `テスト`）もマッピング画面で設定できる。

> ⚠️ **予定時間とポイントについて**  
> 両者は独立して管理する。進捗率の計算はポイントを使用し、開発効率の計算は予定時間・実施時間を使用する。

### 2.3 プロジェクト管理機能

#### 2.3.1 プロジェクト一覧

- 登録済みプロジェクトをカード形式で一覧表示
- 各カードに「現在のイテレーション進捗」をサマリー表示
- 新規プロジェクト作成 / 編集 / 削除
- サイドバーの「プロジェクト登録」メニューは管理者以上（`admin` / `superAdmin`）の権限ユーザーのみに表示する

#### 2.3.2 プロジェクト設定

| 項目 | 説明 |
|------|------|
| プロジェクト名 | 任意の名称 |
| データソース種別 | **GitHub Projects API** / **CSV** から選択 |
| GitHub Projects 接続情報 | データソースが API の場合のみ。Project URL または `owner/repo` + Project 番号 + 認証情報 |
| 参照イテレーションカレンダー | 全社共通の年度カレンダーを選択（§2.4.1）|
| 開始イテレーション | カレンダー上のどのイテレーションから参加するかを選択 |
| プルリク中を進捗に含める | ON / OFF で切り替え（プロジェクト単位）。ON の場合「プルリク中」ステータスのタスクを完了と同じ扱いで進捗率に算入、OFF の場合は未着手と同じ扱いで進捗率に算入しない |
| 総工数（FE / BE / テスト） | データソース同期時に区分ごとのポイント合計を自動集計して表示。手動編集不可（再同期で更新）|

##### 総工数の集計仕様

データソース同期時（API 取得 / CSV インポート）に以下を自動集計し、プロジェクト基本情報に表示する。

| 集計単位 | 内容 |
|---------|------|
| プロジェクト全体 | FE / BE / テスト それぞれの総ポイント合計 |
| イテレーションごと | 各イテレーション内の FE / BE / テスト それぞれのポイント合計 |

```
例）プロジェクト基本情報の総工数表示

  総工数（全体）
  ┌──────┬──────┬────────┬────────┐
  │      │  FE  │   BE   │  テスト │
  ├──────┼──────┼────────┼────────┤
  │ 全体 │ 80pt │  60pt  │  30pt  │
  │ IT#1 │ 20pt │  15pt  │   8pt  │
  │ IT#2 │ 25pt │  20pt  │  10pt  │
  │ IT#3 │ 35pt │  25pt  │  12pt  │
  └──────┴──────┴────────┴────────┘
```

### 2.4 イテレーション管理機能

#### 2.4.1 イテレーションカレンダー（全社共通マスタ）

イテレーションの期間は **開発者・管理者が事前登録する「イテレーションカレンダー」から取得** する。
プロジェクト個別の自動算出は行わない。

- 開発者・管理者が **年度単位で 1 年分のイテレーション期間を一括登録** する
- イテレーション期間は **原則 2 週間** だが、年度開始・年末年始・大型連休・組織イベント等を踏まえ **変則的な期間（短縮 / 延長）も許容** する
- カレンダーは **全プロジェクト共通**。各プロジェクトはカレンダーを参照する（プロジェクトごとの個別期間設定は行わない）
- 各イテレーションは「番号」「開始日」「終了日」を持つ

##### 画面（開発者・管理者専用）

| 項目 | 説明 |
|------|------|
| 年度 | 例：`2026年度` |
| イテレーション一覧 | 番号 / 開始日 / 終了日 / 備考（変則理由など）|
| 操作 | 行追加・編集・削除（変則期間の理由をメモ可）|

**アクセス経路：** `設定 > イテレーションカレンダー設定`  
**操作権限：** 開発者・管理者のみ

#### 2.4.2 プロジェクトとイテレーションの紐付け

- プロジェクト設定で **そのプロジェクトの開始イテレーション**（カレンダー上のどのイテレーションから参加するか）を選択する
- 開始イテレーション以降のすべてのイテレーションが、そのプロジェクトの対象となる
- イテレーション単位で進捗データを表示・切り替え可能
- 過去のイテレーションも参照できる（履歴表示）

### 2.5 チームメンバー管理機能

#### 2.5.1 メンバー情報（イテレーションごとに設定）

| 項目 | 入力方法 | 説明 |
|------|----------|------|
| メンバー名 | 手入力 | 氏名またはアカウント名 |
| 区分 | 選択（FE / BE） | 担当領域 |
| 稼働日数 | 手入力（数値） | そのイテレーション中の稼働日数（詳細は 2.5.2 参照）|
| 目標ポイント | 手入力（数値） | そのイテレーションでの目標ストーリーポイント |
| ベロシティ（実績） | 自動計算 | メンバー個人の消化ポイント ÷ 稼働日数 で自動計算 |
| 目標ポイント | 自動計算 / 手入力 | 稼働日数 × 計画ベロシティ で自動計算。手動上書き可 |

#### 2.5.2 稼働日管理

**アクセス場所：** プロジェクト設定画面内の「稼働日管理」タブ

##### デフォルト稼働日

- 会社全体で 1 つの「イテレーションあたりデフォルト稼働日数」をシステムで固定設定する
- 例：2 週間イテレーションで祝日なしの場合 → `10日`
- このデフォルト値は全プロジェクト・全メンバー共通で適用される

##### 稼働日の個別変更（メンバー × イテレーション単位）

各メンバーの稼働日数はイテレーションごとに個別変更できる。  
有給取得・アサイン率変更・病欠などプロジェクトごとの事情に対応する。

| 項目 | 説明 |
|------|------|
| 稼働日数 | デフォルト値を初期表示。手動で上書き可能 |
| 変更フラグ | デフォルトから変更された場合、**値を赤字で表示** |
| 変更メモ | 変更理由などを自由記述（例：「有給3日取得」「アサイン率50%のため」）|

##### 画面イメージ（稼働日管理テーブル）

```
イテレーション #3  （2026/04/01 〜 2026/04/14）  デフォルト稼働日: 10日

┌──────────┬──────┬──────────────────┬──────────────────────────┐
│ メンバー │ 区分 │ 稼働日数         │ 変更メモ                 │
├──────────┼──────┼──────────────────┼──────────────────────────┤
│ 田中     │ FE   │ 10               │                          │
│ 鈴木     │ FE   │ 🔴 7             │ 有給3日取得              │
│ 佐藤     │ BE   │ 10               │                          │
│ 山田     │ BE   │ 🔴 5             │ 他PJアサイン50%のため    │
└──────────┴──────┴──────────────────┴──────────────────────────┘
```

> - デフォルト値から変更された稼働日数は **赤字** で表示する
> - 変更メモは任意入力。変更した場合に理由を残すことを推奨する
> - 過去イテレーションの稼働日・メモも参照可能（読み取り専用）

#### 2.5.3 個人設定

ログインユーザー自身のアカウント情報を確認・編集するための画面を提供する。

**アクセス経路：** ヘッダーメニュー `個人設定`（`/profile`）

| 項目 | 表示 | 編集可否 | 説明 |
|------|------|----------|------|
| GitHub Account Name | 必須 | 不可 | GitHub ログイン時に取得したアカウント名 |
| 表示名 | 必須 | **可** | アプリ内表示用の名前。ユーザー本人が変更可能 |
| GitHub Account ID | 必須 | 不可 | GitHub アカウント識別子 |
| 権限 | 必須 | 不可 | `superAdmin` / `admin` / `user` のいずれかを表示 |

> ⚠️ 個人設定画面では **表示名のみ編集可能** とし、他項目は参照専用とする。

### 2.6 進捗表示機能

#### 2.6.1 進捗カード（メイン表示）

各プロジェクト × 区分（全体 / FE / BE）ごとに進捗カードを表示する。

```
┌─────────────────────────────────────────────┐
│  Project A  [全体]              2026 IT#3   │
│  進捗  65%  [██████████░░░░░]               │
│  予定 70%   実績 65%                        │
│  完了: 13pt / 目標: 20pt                    │
└─────────────────────────────────────────────┘
```

| 表示項目 | 説明 |
|---------|------|
| プロジェクト名 + 区分 | カードヘッダー |
| 進捗バー | 実績ポイント / 目標ポイント の割合（%）|
| 予定 % | 経過日数から算出した理想進捗率 |
| 実績 % | 完了ポイント / 目標ポイント |
| 完了 pt / 目標 pt | 数値表示 |

#### 2.6.2 横断ビュー

- 全プロジェクトの進捗カードを一画面に並べて表示
- フィルター：区分（全体 / FE / BE）、イテレーション番号
- ソート：プロジェクト名 / 進捗率

#### 2.6.3 詳細ビュー（プロジェクト個別）

- 全体 / FE / BE の 3 カードを並列表示
- メンバーごとの完了ポイント一覧
- メンバー一覧に以下を表示する：`メンバー名 / 区分 / 稼働日数 / 計画ベロシティ（目標ベロシティ） / ベロシティ（実績） / 開発効率 / 目標ポイント`
- メンバー一覧はイテレーション切替で表示対象を変更できる
- 進捗カードの集計範囲は、イテレーション切替とは別に `選択のみ` / `選択まで累計` を切り替え可能にする
- `選択のみ` は選択中イテレーション単体の値を表示する
- `選択まで累計` は開始イテレーションから選択中イテレーションまでを集計した値を表示する
- 切替対象は「過去イテレーション」と「現在イテレーション」のみ。次イテレーション以降は選択不可（無効表示）
- 稼働日テーブルは「各個人の稼働日数を調整するテーブル」とし、`メンバー名 / 区分 / 稼働日数` を表示する
- イテレーション切替UIには各イテレーションの `開始日 / 終了日` を表示する

##### 閲覧権限（詳細ビュー）

- `user` は **自分が所属しているプロジェクトのみ** 詳細画面を閲覧可能
- `user` が未所属プロジェクトへ直接 URL 遷移した場合は、詳細情報を表示せずアクセス拒否とする
- `admin` / `superAdmin` は全プロジェクト詳細を閲覧可能

##### ユーザー別の過去情報閲覧

- `user` は **自分自身の過去情報**（過去イテレーションの実績・稼働情報）を閲覧可能
- `admin` / `superAdmin` は **各ユーザーの過去情報** を閲覧可能
- 導線は「ユーザー一覧から遷移」または「サイドメニューから選択」のいずれかで提供する（方式は §8 で確定）

##### 開発効率（イテレーション別）

| 指標 | 定義 | 解釈 |
|------|------|------|
| 開発効率 | 実施時間合計 ÷ 予定時間合計 | 1.0 が予定通り。1.0 超で予定より時間がかかっている |

##### 進捗推移・予想グラフ

折れ線グラフで以下の 3 本の線を表示する。

| 線の種類 | 説明 | 表示区間 | 線種 |
|---------|------|---------|------|
| 実績進捗率 | 各イテレーション終了時点の実際の進捗率 | 過去〜現在 | **実線** |
| 理想進捗率 | イテレーション数に応じた均等な理想ライン | 全期間 | **実線** |
| 予想進捗率 | 全イテレーション（過去〜現在）の平均ベロシティをもとに算出した予測ライン | 現在〜将来 | **点線** |

**予想進捗の計算方法**

```
平均ベロシティ = 全イテレーション（過去〜現在）の完了ポイント合計 ÷ イテレーション数
予想完了ポイント（次イテレーション以降）= 現在の完了ポイント + 平均ベロシティ × 残イテレーション数
予想進捗率 = 予想完了ポイント ÷ 目標ポイント合計
```

予想進捗率の算出は、常に「全イテレーション（過去〜現在）」を母数として行う。
