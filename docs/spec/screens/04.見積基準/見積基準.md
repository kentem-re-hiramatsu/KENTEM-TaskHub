# 見積基準

## 概要

ポイント換算基準・工数換算基準の参照・管理を行う（2nd 以降対応）。

**アクセス経路：** サイドバー「見積基準」

---

## 権限

| 権限 | アクセス |
|------|---------|
| `superAdmin`（開発者）| 表示・操作可能 |
| `admin` / `user` | 表示しない |

---

## 仕様

- 2nd フェーズ以降に詳細仕様を策定する
- ポイント・工数の換算ルールをどこまで設定可能にするかは未決（[未決事項](../index.md#未決事項) 参照）
- 見積レビュー会での活用を想定している

---

## 検討事項

### フロー

- 見積基準を参照・更新する際の全体的なフロー（誰が・いつ・どの手順で行うか）を決める必要がある
  - 例：担当者が申請 → 承認者が承認 → 見積基準に反映、のような承認フローが必要か
  - 更新頻度・タイミング（年次更新 / 都度更新）はどうするか

### 申請方法

- 見積基準の変更・追加を申請する手段を検討する
  - 画面上からフォーム入力で申請するか、別途申請フォーム（メール・チケット等）を使うか
  - 申請できるロールを限定するか（例：`admin` のみ申請可能）

### 見積基準担当者への通知方法

- 申請が来た際に、見積基準を管理する担当者へどう通知するかを検討する
  - 通知手段：メール / システム内通知（アプリ内バナー・ベル通知）/ Slack 連携 など
  - 通知タイミング：申請時 / 差し戻し時 / 承認完了時

### UI（入力の手間を減らす）

- 入力項目が多いため、ユーザーの入力負荷を下げるUI設計を検討する
  - **一括入力・インポート**：CSVやExcelからの一括取り込みで手入力を最小化する
  - **デフォルト値・テンプレート**：よく使う基準値をテンプレートとして保存・呼び出せるようにする
  - **コピー機能**：既存の見積基準をベースに複製して編集できるようにする
  - **インライン編集**：一覧画面上で直接セルを編集できるようにし、詳細画面への遷移を減らす
  - **自動補完・サジェスト**：過去の入力値や関連マスタから候補を提示する
  - 上記のうちどの手段を採用するかは優先度・工数を踏まえて決定する

### 修正指摘方法（赤ペン・コメント機能）

- 見積基準の内容に対して修正を指摘する手段を検討する
  - **コメント（メモ）方式**：各項目・セルに対してコメントを紐付けて残す（Excel のコメント機能に近いイメージ）
  - **赤ペン方式**：テキスト上に直接マーキング・打ち消し線・上書きなどの注釈を入れられるようにする
  - **差し戻しコメント方式**：申請を差し戻す際に理由・修正箇所をフリーテキストで入力する
  - どの粒度でコメントを紐付けるか（行単位 / セル単位 / 申請全体）を決める必要がある

### 修正後の変更箇所の可視化

- 修正前後の差分を分かりやすく表示する方法を検討する
  - **GitHub PR の diff 表示を理想形とする**：変更前（赤・削除）／変更後（緑・追加）を並べて表示する
  - 差分表示の対象：数値・テキストの変更 / 行の追加・削除
  - 履歴として過去バージョンを遡って比較できるようにするか
  - 差分表示は申請レビュー時だけでなく、承認済み履歴でも参照できるようにするか
